compiledRelease { }

Top-level object returned by the FTS OCDS API. Represents the current compiled state of a single contracting process identified by its ocid. All notices published for the same procurement share the same OCID and are merged into this single document. OCID format: ocds-h6vhtk-[0-9a-f]{6}

parties [ ] array (max 102,000) · object · All organisations referenced in this process
▼
All organisations referenced anywhere in the contracting process are consolidated here — buyers, suppliers, tenderers, procuring entities. Each entry has a roles[] array defining its function.

⚠ Non-exclusive roles: an organisation can appear multiple times with different roles. A framework owner may appear as both buyer and procuringEntity. Deduplication logic is required for spend and supplier analysis. Organisation ID format: parties/identifier/scheme + - + parties/identifier/id. Supported schemes: GB-COH, GB-CHC, GB-SC, GB-NIC, GB-NHS, GB-UKPRN, GB-MPR, GG-RCE, JE-FSC, IM-CR, GB-PPON

└ identifier/idstringIdentifier value within the scheme (e.g. Companies House number).

buyer { } object · Reference to the contracting authority in parties[]
▼
⚠ Overwrite semantics: the buyer field reflects the most recent notice published. If a Machinery of Government change occurs mid-procurement, original buyer attribution is lost. Use the full record releases[] to reconstruct buyer history. A shallow reference only — the full organisation detail is in parties[] matched by buyer/id.
buyer/idstringMatches a parties/id with role buyer. Format: scheme-id (e.g. GB-GOR-CO)

awards [ ] array (max 10,000) · object · PA23 notices: UK5 (award), UK6 (transparency / direct award)
▼
✓ Strong data quality. Award notices are mandatory under PA23 and contain supplier identity (including SME/VCSE via parties[]), award value, contract period and lot references. Primary source for spend and supplier analytics. One array entry per award decision. A multi-lot procurement may have many awards entries. Positive id = pending/active; negative = cancelled.

contracts [ ] array (max 10,000) · object · PA23 notices: UK7 (signed), UK10 (amendment), UK11 (termination)
▼
Data maturity: contract management data is the weakest part of the PA23 dataset. KPI reporting (agreedMetrics[]) is mandated but regulation permits KPI definitions to change at each reporting cycle — time-series analysis is structurally difficult. implementation is present but sparse. Section 70 payment data will eventually flow via a separate mechanism — not yet in this schema. Links back to awards[] via contracts/awardID. One contract entry per signed contract (may not equal one per award in multi-lot scenarios).
⚠ Below-threshold closure: UK11 (termination notice) is not mandatory for below-threshold contracts. If today > contracts/period/maxExtentDate and no termination notice exists, status should be inferred as "below-threshold assumed completed".

relatedProcesses [ ] array (max 2) · object · Links to other contracting processes
▼
Links this contracting process to related OCIDs — typically the parent framework used for a call-off, a prior open framework, or an unsuccessful competitive process being switched to direct award. Coverage is inconsistent — frameworks should link back via relatedProcesses, but this varies by contracting authority.
