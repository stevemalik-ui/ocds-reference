OCDS Compiled Release Structure

Find a Tender Service — Complete Data Reference

compiledRelease {}
The top-level object returned by the OCDS API. Contains the current state of a single contracting process (OCID). Merges all notices (releases) into a single view. Array fields contain 0 or more items.
Core Identifiers & Metadata ▼
ocid string
Open Contracting ID. Unique identifier for the contracting process. Prefix: ocds-b5fd17- followed by an 8-character identifier.
Critical for analysis: The OCID is the spine of the dataset. All notices, awards, and contracts for a single procurement share the same OCID. Use this to aggregate across notices and reconstruct timelines.
```
id string
Release ID. Identifier for this compiled release (usually same as ocid, or may be versioned).
date string (ISO 8601)
Date when this release was published. Format: YYYY-MM-DDTHH:MM:SSZ
version string
OCDS version (e.g., "1.1.5"). Indicates which version of the OCDS standard was used.
```
buyer {} — Contracting Authority ▼
⚠️ Overwrite semantics: The buyer field in a compiled release reflects the most recent notice published. If a Machinery of Government change occurs mid-procurement, you lose the original buyer attribution. Always pull the full record to reconstruct buyer history from releases[].
```
id string
Org ID. Format: GB-GOR-CO (Cabinet Office), GB-NHS-XXXX, GB-COH-12345678 (Companies House).
Entity resolution challenge: No single canonical org ID. Same org may appear under different ID schemes across notices. Requires external matching against Companies House or GOV.UK org hierarchy.
name string
Legal name of the contracting authority (e.g., "Cabinet Office", "NHS England").
details.scale string
"central" | "regional" | "local" — Buyer size classification (rarely populated for buyers; mainly used for suppliers).
```
tender {} — Procurement Details ▼
id string
Tender ID. Identifier for the procurement procedure within the contracting authority's system.
```
title string
Plain-language title of the procurement (e.g., "Consultancy Services for Digital Transformation").
description string
Detailed description of what is being procured. Often lengthy free text.
status string
Current procedure status: planning | tender | awarded | complete | withdrawn | cancelled
Below-threshold caveat: For below-threshold procurements, no closure notice is mandated. Status must be inferred from dates. See contracts[].period.endDate.
procurementMethod string
How the procurement was conducted: open | limited | selective | direct
value object
Estimated procurement value (buyer's budget).
amount number
Numerical amount (e.g., 500000)
currency string
ISO 4217 code (e.g., "GBP")
lots[] array object
Subdivisions of the procurement. Each lot can have separate suppliers and values. A procurement may be competed as a single lot (length = 1) or multiple lots (length > 1).
id string
Lot identifier (e.g., "lot-1")
title string
Plain-language lot name
value object
Lot-level budget {amount, currency}
items[] array object
Line items being procured. Each item has a CPV (Common Procurement Vocabulary) classification.
id string
Item identifier
description string
What is being bought
classification object
{scheme: "CPV", id: "73200000", description: "..."}
CPV is the EU procurement classification. 8-digit code. Used for filtering and spend analysis by category.
tenderPeriod object
When suppliers can submit proposals. {startDate, endDate} in ISO 8601 format.
documentDeadline string
When supporting documentation must be submitted (if different from endDate).
hasEnquiries boolean
Whether clarification questions were received from potential suppliers.
awardCriteria array
How the winner will be selected (e.g., "lowest cost" vs "most economically advantageous tender").
```
awards[] array— Contract Awards ▼
id string
Award identifier (e.g., "award-001"). May correspond to a lot or represent a standalone award in an open procedure.
```
status string
pending | active | cancelled — Current state of the award.
date string (ISO 8601)
Date the award decision was made (contract signature date or award notification date).
value object
Contract award value. {amount, currency}. The actual value paid to the supplier(s).
suppliers[] array object
List of organisations that won this award. May be 1 (sole supplier) or >1 (consortium/framework).
id string
Supplier organisation ID. Format: GB-COH-12345678 (Companies House), GB-SIRL-XXXXX, or free text.
name string
Legal name of the supplier.
contractPeriod object
Planned or actual contract duration. {startDate, endDate, durationInDays (optional), maxExtentDate (optional)}
Inference of closure: For below-threshold contracts with no termination notice, if today > maxExtentDate, status should be inferred as "assumed completed".
contractID string
Reference to a contracts[] entry (by contracts[].id). Links award to implementation.
amendment object
Changes to the award post-signature (scope changes, price changes, extension). {date, rationale, description}
```
contracts[] array— Contract Implementation ▼
id string
Contract identifier (e.g., "contract-001"). Should correspond to a contracts[].id referenced by awards[].contractID.
```
awardID string
Reference back to the award (awards[].id) that this contract implements.
status string
pending | active | terminated | complete — Implementation status.
period object
Contract duration. {startDate, endDate, durationInDays (optional), maxExtentDate (optional)}
Data quality: startDate and endDate are often sparse, especially for legacy contracts. maxExtentDate (the latest possible end including extensions) is more reliable for status inference.
value object
Contract value. May differ from awards[].value if scope or price changed post-award.
implementation object
Emerging data: Contract performance tracking and Section 70 payment transparency. Currently sparse but growing under PA23.
milestones[] array object
Project milestones and deliverables. Rarely populated. {id, title, description, dueDate, status}
documents[] array object
Links to contract documents (PDF specifications, SLAs, etc.). Mostly absent in current data.
transactions[] array object
Section 70 Payment Data (Coming 2025): Payment records will be published here. {id, date, amount, payer, payee, status: "pending|successful|failed"} — Links invoice dates to payment dates for prompt-pay compliance tracking.
metrics[] array object
KPI/performance metrics reported against the contract. {id, title, value, period}.
Data quality warning: The regulation permits KPI definitions to change at each reporting cycle — makes time-series comparison very difficult.
amendment object
Changes to contract post-signature. {date, changes (free text), rationale}
```
parties[] array— All Organisations ▼
⚠️ Non-exclusive roles: An organisation can appear multiple times in parties[] with different roles. A framework owner may appear as both "buyer" and "procuringEntity". A supplier may appear in multiple awards with different roles. Deduplication logic required for spend and supplier analysis.
```
id string
Organisation ID. Same format as buyer.id (GB-GOR, GB-NHS, GB-COH, etc.).
name string
Legal name of the organisation.
roles[] array string
What function(s) the organisation performed in this process. Possible values:
  • buyer — Contracting authority
  • tenderer — Submitted a proposal
  • supplier — Won a contract
  • procuringEntity — Entity for which the buyer is buying (often same as buyer)
  • payee — Receives payment
  • payer — Makes payment (government entity)
details object
Additional organisation attributes.
scale string
"sme" | "large" | "vcse" | "micro" | "startup" — Size/type classification of the organisation. Critical field for SME/VCSE spend analytics.
website string
Organisation website URL (rarely populated).
address object
Registered address. {streetAddress, locality, region, postalCode, countryName}
contactPoint object
Contact details for the organisation. {name, email, telephone, faxNumber, url}
```
planning {} — Pre-Market & Budget ▼
budget object
Budget allocated for the procurement. {amount, currency} — Often sparse. Partially overlaps with tender.value.
```
rationale string
Why the procurement is being conducted (business justification). Optional, rarely populated.
documents array
Links to business case or strategy documents. Mostly absent.
milestone object
Planned procurement milestone dates. {description, dueDate, status} — Often sparse.
Data maturity: The planning block is structurally sound but has very low coverage under PA23. As contracting authorities adopt planning notices, this will become richer. Currently not a reliable source for spend forecasting.
```
relatedProcesses[] array— Procurement Links ▼
id string
Identifier for the related process entry.
```
relationship string
Nature of the relationship: parent | child | renewal | tender (etc.)
relatedProcessID object
Reference to another OCID. {scheme: "OCDS", id: "ocds-b5fd17-00000999"}
Use case: Link framework calls-off to the parent framework OCID, or link a renewal procure to its predecessor. Currently sparse — frameworks should link back via relatedProcesses, but coverage varies.
```
Object {} — Single nested structure
Array [] — 0, 1, or many items
String/Scalar — Primitive data type
